# Recruitcha — Completed OS Example

This folder contains a **fully filled example** of every `company/*.md` file, demonstrating what a completed Client OS looks like after onboarding.

**Use this when:**
- Onboarding a new client and unsure how detailed each section should be
- Reviewing the quality bar for "completed" company files
- Training Claude on what good looks like

**Do NOT:**
- Copy these files verbatim — they're specific to Recruitcha
- Reference Recruitcha-specific claims in other clients' copy
- Edit these — they're frozen examples

## Files

| File | What it shows |
|------|--------------|
| `_config.md` | Central config — all key client metadata in one place |
| `overview.md` | Full client identity, SLA, success criteria, context notes |
| `icp.md` | 3-layer ICP, signal priority with PRR data, avoid list, lead filters |
| `offer.md` | One-line offer, proof points with named clients and verified figures, differentiators, value props |
| `voice.md` | Tone, banned words, approved claims (curated), industry vernacular |
| `campaign-state.md` | Active + paused campaigns, tools, infrastructure, benchmarks |

## What makes these "good"

1. **Specific numbers everywhere.** No "many clients" — "70+ funded startups placed with."
2. **Named case studies.** TurboTenant, HealthNote, LSAT Demon — verifiable.
3. **Approved claims are explicit.** No room for Claude to invent.
4. **Decision rationale embedded.** ICP notes explain WHY size was tightened from 500 to 200.
5. **Voice notes include real preferences.** Drew is Texas-based → American English.
6. **Campaign state shows trends.** Not just current metrics — also "declining 4 weeks straight" context.

## How to use

When onboarding a new client and stuck on a section, open the corresponding file here. Match the level of detail. Adapt for your client.

Filled file = ~80% as detailed as Recruitcha's = quality bar met.
Filled file = thin placeholders or vague text = redo with onboarder.
